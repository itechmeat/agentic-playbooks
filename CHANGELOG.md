# Changelog

Format: [Keep a Changelog], versions: semver (major - breaking changes to the playbook.yaml schema or CLI, minor - new features, patch - fixes).

The version stays at `0.1.0` (in development) until the full plan functionality is implemented and tested; the version is not bumped for individual phases.

## [0.1.0] - in development

### Added
- playbook.yaml schema (schema: 1) and validator V01-V15.
- .apb/ directory structure (playbooks/versions/current/layouts/profiles/runs), atomic writes for control files.
- CLI: apb init, apb list, apb validate, apb dashboard, apb run, apb runs, apb resume.
- Web: playbook list, read-only graph (svelte-flow, dagre auto-layout, horizontal layout), live updates from disk (watcher + WebSocket), full-screen layout with a 40px header.
- serve.lock with a root fingerprint and instance id, lock released on graceful shutdown (SIGINT/SIGTERM).
- Execution engine (apb-engine): event sourcing, state folding, retry and fallbacks, sh-script, shared context, one-off instruction, enforced max_loops, immutable version snapshot per run, single-writer lock on the working directory.
- Web run monitor: run list, run page with live highlighting of node statuses and an event feed (watcher on .apb/runs + WebSocket).
- MCP server (apb mcp, stdio): read tools (list/get/validate) and run tools (run/runs/status/events/report/resume) for the coding agent.
- Engine: background (non-blocking) run, run_cancel, a channel for wake events and control commands, a supervised run mode (groundwork for a supervising agent).
- Supervisor MCP tools (supervisor_wait_event/run_inspect/node_retry/run_continue_from/run_pause/run_abort/context_append/report) and supervise:self: the coding agent supervises a run from its own session via a token, using a capability model.
- Background supervising agent (apb run --supervise): the engine spawns the supervising agent, heartbeat and supervisor_lost with a safety-net respawn, auto-summarized report; an intervention log and the supervising agent's report on the run page in the web UI.
- Playbook minor-version machinery: immutable version directories, atomic version-number issuance, copying scripts/ and layout from the parent, a validation gate, soft-delete to trash with restore; structural and text (LCS) version diffs; MCP write tools (playbook_create/update/delete) and HTTP endpoints (POST/PUT/DELETE /api/playbooks, PUT .../layout, GET .../diff).
- Web playbook editor: a YAML panel (source of truth) plus a live graph, a node palette and property forms, edge connection, edits via YAML-AST (preserving all fields), saving as a new minor version, auto-saving layout on drag, a version switcher with diff, CRUD on the list (create/duplicate/delete).
- Patch-version machinery: provenance sidecars, run-migration validation, patch/migration events, a patch limit and policy-driven promotion to an improvement version after a successful run.
- Supervising-agent self-patches: the supervisor_patch_playbook MCP tool (creates a patch version from YAML and migrates the run onto it), gated by the patch_playbook capability (now included in the default capability set); HTTP endpoints GET /api/playbooks/{id}/versions (versions with provenance) and POST .../versions/{version}/promote (manual promotion); a version-history panel in the web UI with provenance and a promote button.
- human_review node: pauses the run until a human or agent decides; the decision is made via `apb review`, the review_decide MCP tool, or HTTP POST /api/runs/{id}/review, plus buttons on the run page in the web UI; the reviewer's note is available as {{nodes.<id>.review_note}}. Branching completeness: the review_status and output_match (substring match) edge conditions are now taken into account when picking the next node.
- wait node: waits for a timer (wait_for timer) or a webhook signal (wait_for webhook), with a mandatory timeout_seconds (timing out fails the node). For each webhook key, an unpredictable hook_secret (uuid v4) is generated at run start; the signal is accepted via HTTP POST /api/hooks/{run-id}/{secret}, bound to the run; the endpoint path is available as {{run.hooks.<key>}} and is shown on the run page in the web UI with a waiting indicator.
- Parallel branches: multiple unconditional outgoing edges from a node mean parallel branches; convergence happens at a node with a join field on incoming edges (join: all - wait for all branches, the default; join: any - continue on the first, cancel the rest); reaching finish in one branch completes the run and cancels the others. The engine has moved to an active-node-frontier model; linear runs, the supervisor, patch, human_review, and wait are unchanged. Slow nodes in different branches (agent_task/script) execute concurrently on threads, while only drive writes events (order = completion order). On join:any, the engine kills the losing branch's process (agent_task).
- Phase 8 (agent and runner breadth, reliability): agent_task timeout (timeout_seconds kills the process, status TimedOut, fallback); context compaction with a cheap model into a separate context_compact.md when context_max_bytes is exceeded (the primary context.md is left untouched for replay determinism, the summary is not written to the log); global config ~/.config/apb/config.yaml (agents by id with a program and transport, named executors overridable locally, a runner registry, a default executor, a port); an extensible runner registry (ts:[bun,deno]/py:[python3]/sh plus config, first available on PATH, classified by basename); a pluggable agent transport headless|acp, where acp is streaming stream-json with a per-attempt NDJSON log and error classes. Cancellation/timeout kill the agent's entire process group (Unix process_group + kill(-pgid)).
- Agent report contract (spec 6.2): an agent node's status is taken from the final yaml block (`status: success|failure`) in the response, not from the process return code, so the agent's self-assessment enables branching on node_status. Backward compatible: without the block, the status defaults to success. Plus `success_check` - an sh-script check of an agent node's result; a nonzero exit code marks the node failed regardless of the self-assessment (V12, node form in the web UI).
- Run overrides (spec 11): `apb run --overrides <file.yaml>` runs a playbook with different models/executors without creating a new version. The engine works with an "effective playbook" (version + overrides); the run snapshot reflects the effective playbook, and the overrides are saved in run.yaml.
- Node isolation (8e-2, declarative): an `isolation: full|best_effort|none` field on agent_task, visible on the node form in the web UI; the validator warns (V16) that full/best_effort are declared but not yet enforced by the engine (execution happens in the shared working directory). Actual enforcement is future work.
- Phase 9 (operations and distribution): apb doctor (environment diagnostics: config, registry, agent programs and runner runtimes on PATH, playbook validity, exit code); apb dev (Vite HMR plus an API server for developing from the source tree); apb export / apb import (moving a playbook as a single JSON bundle with editor layout); CI and a release playbook (GitHub Actions: building the web frontend plus macOS/Linux binaries on tag), docs/INSTALL.md, and a brew-formula template packaging/apb.rb.
