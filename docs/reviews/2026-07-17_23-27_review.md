## Executive Summary

The implementation is well-structured and follows the project's conventions closely. The pure fold model for progress computation is deterministic, correctly mirrors `RunState::fold`, and the iterative Tarjan SCC is algorithmically sound. The single-writer invariant on the event log is preserved. The code is production-ready with one notable performance issue in `run_report` (double playbook/events load) and a few minor observations around type safety and missing test coverage.

## Findings

### [MAJOR] Double playbook and events load in `run_report`

**Location:** `crates/apb-mcp/src/tools.rs:748-759`
**Principles:** DRY, Performance

`run_report` calls `run_status` internally, which already:
1. Reads `events.jsonl` via `read_all(&dir)`
2. Loads and parses `playbook.yaml` via `from_run_dir` -> `load_run_playbook`

Then `run_report` immediately calls `load_run_playbook(&dir)` again (YAML parse #2), and `build_duration_table(&dir, &pb)` calls `read_all(run_dir)` again (events read #2).

For a single `run_report` MCP call, the playbook snapshot is parsed from YAML twice and events.jsonl is read from disk twice.

**Suggested fix:**

Refactor `run_status` to accept an optional pre-loaded playbook, or extract the shared logic so `run_report` loads once:

```rust
pub fn run_report(root: &Path, run_id: &str) -> Result<Value, ToolError> {
    let dir = resolve_run_dir(root, run_id)?;
    let events = read_all(&dir).map_err(|e| ToolError::Engine(e.to_string()))?;
    let state = RunState::fold(&events);
    let pb = apb_engine::progress::load_run_playbook(&dir);
    let progress = pb.as_ref().map(|p| apb_engine::progress::compute(p, &events));

    let nodes: BTreeMap<String, String> = state
        .nodes
        .iter()
        .map(|(k, v)| (k.clone(), v.as_str().to_string()))
        .collect();
    let mut base = json!({
        "run_id": run_id,
        "run_status": state.run_status.as_str(),
        "nodes": nodes,
        "outputs": state.outputs,
        "progress": progress,
    });

    if let Some(ref playbook) = pb {
        let measured = apb_engine::progress::node_durations_seconds(&events);
        let table = build_duration_table_from(playbook, &measured);
        if let Some(obj) = base.as_object_mut() {
            obj.insert("duration_table".into(), json!(table));
        }
    }
    Ok(base)
}
```

Where `build_duration_table_from` takes the playbook and a pre-computed `measured` map instead of re-reading events.

### [MINOR] `node_durations_seconds` has no unit test

**Location:** `crates/apb-engine/src/progress.rs:244-261`
**Principles:** Test coverage

The function is `pub`, exported from `apb-engine`, and consumed by `build_duration_table` in the MCP tools layer. It computes wall time per node from NodeStarted/NodeFinished events. It handles loops (last completion wins) and uses `saturating_sub` for safety.

There is no dedicated unit test for this function. The progress fold tests (`compute`) exercise event payloads but not duration measurement. A regression in this function would silently produce wrong calibration data in `run_report` without failing any test.

**Suggested fix:**

Add a test to the `#[cfg(test)] mod tests` block in `progress.rs`:

```rust
#[test]
fn node_durations_measures_started_to_finished() {
    let events = vec![
        Event { seq: 0, ts: 1_000, payload: EventPayload::NodeStarted {
            node: "a".into(), attempt: 1,
        }},
        Event { seq: 1, ts: 4_500, payload: EventPayload::NodeFinished {
            node: "a".into(), status: "succeeded".into(), attempt: 1, output: String::new(),
        }},
    ];
    let d = node_durations_seconds(&events);
    assert_eq!(d.get("a"), Some(&3)); // (4500 - 1000) / 1000 = 3
}
```

### [MINOR] `waiting_kind` is `Option<String>` instead of an enum

**Location:** `crates/apb-engine/src/progress.rs:24`
**Principles:** Type safety, KISS

`ProgressSummary.waiting_kind` carries `"human_review"` or `"wait"` as a free-form string. The values are constrained by the `compute` function (lines 222-232) but not enforced by the type system. A future edit could introduce a typo or an undocumented value.

The web consumer (`RunProgress.svelte` lines 23-29) pattern-matches on exact string values. An undocumented third value falls through to the generic `"waiting"` label, which is safe but could mask a bug.

**Suggested fix:**

Replace `Option<String>` with `Option<WaitingKind>` where:

```rust
#[derive(Debug, Clone, Serialize)]
#[serde(rename_all = "snake_case")]
pub enum WaitingKind {
    HumanReview,
    Wait,
}
```

This gives compile-time exhaustiveness. The JSON output is unchanged (`"human_review"` / `"wait"`). The web consumer's string match continues to work unchanged.

**Trade-off:** This is a type-safety improvement, not a correctness bug. The current code works. If the team prefers strings for cross-language simplicity, the current approach is acceptable as long as the allowed values are documented on the field.

### [MINOR] `list_runs` now loads playbook + folds events for every run

**Location:** `crates/apb-engine/src/scheduler.rs:1144-1155`
**Principles:** Performance

`list_runs` now calls `from_run_dir` for each run entry, which loads and parses `playbook.yaml` plus folds all events. Previously it only read lightweight metadata.

The runs list is refetched on every `runs_changed` broadcast (file watcher fires on any run dir change). For a project with many runs (e.g. 50+ historical runs), each refetch parses 50+ YAML files and folds 50+ event logs.

**Assessment:** Acceptable for the current use case (local-first tool, typical projects have <50 runs). The spec explicitly says "Live updates require nothing new: appending the event touches the run dir, watch.rs broadcasts runs_changed, the dashboard refetches." The trade-off is simplicity (no caching layer) vs. scalability.

**Suggested mitigation (if needed later):**

Cache the `ProgressSummary` per run dir, invalidated by the event count or last-event-seq. Or compute progress lazily (only for non-terminal runs, since terminal runs show status only). For now, no action needed.

### [MINOR] `load_run_playbook` silently collapses missing and unparseable snapshots

**Location:** `crates/apb-engine/src/progress.rs:88-99`
**Principles:** Observability, fail-loudly

The function returns `None` for both a missing `playbook.yaml` and an unparseable one. The inline comment (lines 90-95) documents this as deliberate: "apb-engine has no tracing/log facility and we do not add a dependency just for this one branch."

The risk: a future schema migration that doesn't update existing run snapshots would silently produce `None` (no progress) for all old runs, with no diagnostic signal. The operator sees "no progress bar" but no error.

**Assessment:** The reasoning is sound for the current state. Run snapshots are immutable and engine-written, so an unparseable snapshot implies filesystem corruption or a version mismatch, both of which are rare. The `load_run_playbook` comment is thorough.

**Suggested mitigation:** Write a one-line `eprintln!` to stderr on the unparseable branch. It costs nothing, requires no dependency, and gives operators a signal in the terminal without adding structured logging. Example:

```rust
pub fn load_run_playbook(run_dir: &Path) -> Option<Playbook> {
    let yaml = std::fs::read_to_string(run_dir.join("playbook.yaml")).ok()?;
    Playbook::from_yaml(&yaml).unwrap_or_else(|e| {
        eprintln!("apb: warning: run snapshot {} unparseable: {e}", run_dir.display());
        None
    })
}
```

### [NIT] `review_pending` closure scans events twice per human_review node

**Location:** `crates/apb-engine/src/progress.rs:207-221`
**Principles:** KISS, Performance

The closure counts `ReviewRequested` events and `ReviewDecided` events in two separate `.iter().filter().count()` passes over the full event list. For each human_review node in `playbook.nodes`, `review_pending` is called.

Since `find_map` (line 222) stops at the first match, the worst case is: all human_review nodes are NOT pending, each scanning events twice. For a playbook with N human_review nodes and E events, that is O(2 * N * E).

**Assessment:** Negligible for typical playbooks (1-3 human_review nodes, <1000 events). Not worth optimizing unless profiling shows it matters. A single-pass counter that tracks all nodes at once would be the refactor if needed.

### [NIT] Duplicated RunProgress append logic between `drive` and `await_control`

**Location:** `crates/apb-engine/src/scheduler.rs:576-584` and `crates/apb-engine/src/scheduler/supervisor.rs:127-136`
**Principles:** DRY

The `Control::Progress` match arm constructs the same `EventPayload::RunProgress` in two places:
1. In `drive`'s top-level control drain loop (scheduler.rs:576)
2. In `await_control`'s in-place handler (supervisor.rs:127)

Both stamp the current node and append the event. This mirrors the existing `ContextAppend` pattern (which is also duplicated), so it is consistent with the codebase convention.

**Assessment:** The duplication is intentional defensive coding (the drive branch is documented as "defensive, in case of future refactoring"). Extracting a helper would add indirection for a 7-line match arm. Acceptable as-is.

## Positive Observations

1. **Pure fold model is the right call.** Computing progress from persisted events (no mutable state) means resume, server restart, and a second machine all show the same number. This is the single most important architectural decision and it is correct.

2. **Single-writer invariant preserved.** The MCP tool posts a `Control::Progress` command; only `drive` appends to `events.jsonl`. This follows the established `ContextAppend` and `Patch` patterns exactly.

3. **Iterative Tarjan SCC is correct.** The implementation handles empty graphs, single nodes, self-loops, and disconnected components correctly. The `expect("frame exists")` at line 59 is safe (the while condition guarantees a non-empty call stack). Using SCCs to detect cycles is cleaner than ad-hoc back-edge detection.

4. **`load_run_playbook` consolidation in server.** The `get_run_handler` refactor in `crates/apb-server/src/lib.rs` replaced inline YAML loading with a call to `progress::load_run_playbook`. This eliminated duplicated parsing logic and kept the loaded playbook in scope for both JSON serialization and progress computation. Good DRY improvement.

5. **`DEFAULT_TASK_SECONDS` is a single named constant.** The 120s default for agent_task/script nodes lives in `duration.rs:6` and is referenced everywhere via `crate::duration::DEFAULT_TASK_SECONDS`. No magic number duplication.

6. **All new `EventPayload` fields use `#[serde(default)]`.** Complies with the AGENTS.md house rule. Old event logs read unchanged.

7. **Svelte 5 runes used correctly.** `RunProgress.svelte` uses `$effect` with `untrack` for reading previous state without creating reactive dependencies. The `$derived` declarations for `visible`, `waiting`, `waitingText` are idiomatic.

8. **Accessibility.** The progress bar has `role="progressbar"`, `aria-valuenow`, `aria-valuemin`, `aria-valuemax`, and `aria-label`. Good.

9. **Monotonicity logic is clean.** `displayPercent` in `progress.ts` implements the spec's rule: percent never decreases within the same plan, with an honest-drop exception when the label changes (signaling a plan change). Well-tested in `progress.test.ts`.

10. **Waiting detection is well-reasoned.** The comment at progress.rs:201-206 explains why `Running` status misses the human_review window (NodeStarted fires only when the decision arrives), justifying the request/decided count approach. The wait node correctly uses Running (it does reach that status while blocking).
