# Review: run progress

Дата: 2026-07-17 19:50 Europe/Belgrade

Статус: не готово к merge

Область ревью: staged, но незакоммиченные изменения относительно `HEAD` из `git diff --cached HEAD`. Исходный код, Git index, `HEAD` и ветки в ходе ревью не изменялись. Коммиты не создавались.

## Итог

Концепция фичи рабочая: progress вычисляется из persisted events и snapshot playbook, единый engine fold переиспользуется HTTP и MCP слоями, а новый mutable state не вводится. Однако в текущей реализации подтверждены пять Important-дефектов корректности. Они могут приводить к откату процента, неверной привязке agent report, отсутствию progress у поддерживаемых legacy runs и недостоверному состоянию progress bar в web UI.

Critical-дефектов не найдено. Исправление Important-пунктов обязательно до merge.

## Что сделано хорошо

- `DEFAULT_TASK_SECONDS` определён в одном месте. Дублирующих backend hardcode со значением `120` не найдено.
- Progress является pure fold по persisted events и snapshot playbook. Отдельный mutable progress store не введён.
- Single-writer invariant сохранён: MCP создаёт `Control::Progress`, а события записывает drive loop.
- Новые поля `EventPayload::RunProgress` снабжены `#[serde(default)]`.
- MCP `run_status`, HTTP list/detail и web используют общий `ProgressSummary`, а не независимые формулы.
- Optional/null поля сохраняют совместимость старых API consumers.
- В backend есть полезные тесты weights, skip, clamp, wait/human review и terminal 100 percent.
- В UI terminality централизована, progressbar имеет ARIA role/value/label, а waiting copy различает semantic node kind.
- Code-ranker не обнаружил архитектурных или complexity нарушений.

## Important findings

### 1. Progress цикла откатывается при начале следующей итерации

Файлы:

- `crates/apb-engine/src/progress.rs:109`
- `crates/apb-engine/src/progress.rs:166`
- `crates/apb-engine/src/state.rs:106`

`progress` начисляет report-free SCC credit только узлам, чей текущий `NodeStatus` равен `Succeeded`. После следующего `NodeStarted` для того же узла `RunState::fold` заменяет `Succeeded` на `Running`, и уже заработанный вклад исчезает.

Трасса: `NodeFinished(work, succeeded) -> progress includes work -> back edge -> NodeStarted(work) -> status Running -> progress drops`.

Это нарушает design contract: завершённый проход цикла должен оставаться зачтённым, а процент не должен уменьшаться без изменения work plan. Локальная UI monotonicity может временно скрыть проблему, но после reload или restart откат снова виден.

Рекомендация: для SCC без explicit report вычислять one-pass credit по истории успешных `NodeFinished`, с cap в один проход. При этом необходимо сохранить корректную обработку latest `Skipped` и `Cancelled`. Добавить regression test `Succeeded -> loop re-entry NodeStarted`.

### 2. `Control::Progress` может быть записан от имени successor node

Файлы:

- `crates/apb-engine/src/scheduler.rs:579`
- `crates/apb-engine/src/scheduler.rs:1092`
- `crates/apb-engine/src/scheduler/supervisor.rs:111`

При proactive drain событие получает `node_id: current`. Команда, отправленная агентом во время `execute_node` узла A, обычно читается только на следующем top-of-loop. До этого scheduler успевает заменить `current` на successor B через `frontier.remove(0)`. В результате report узла A сохраняется как `RunProgress { node_id: B }`.

Это особенно опасно на выходе из SCC и в parallel execution: report масштабируется другой группой либо игнорируется формулой.

Минимальное исправление: дренировать progress commands сразу после `execute_node` и до frontier advancement, пока `current` всё ещё равен A. Надёжное исправление для concurrent reporters: включить validated `node_id` в control command и не выводить identity из неявного `current`. Нужен drive-level integration test полного пути `Control::Progress -> scheduler -> RunProgress` с проверкой `node_id`.

### 3. Поддерживаемые legacy snapshots получают `progress: null`

Файлы:

- `crates/apb-engine/src/progress.rs:96`
- `crates/apb-core/src/schema.rs:82`
- `crates/apb-engine/src/scheduler.rs:1219`

`load_run_playbook` использует строгий `Playbook::from_yaml`, который намеренно возвращает `LegacyExecutors`. При этом resume path распознаёт legacy snapshot и парсит его через compatibility branch. Один и тот же run может успешно resume, но list/detail/MCP возвращают `progress: null`, хотя design требует fallback calculation по per-kind defaults.

Рекомендация: вынести единый read-only parser для persisted run snapshots, который повторяет compatibility semantics resume path, не ослабляя строгий parser live definitions. Переиспользовать его в progress/report consumers и добавить тест schema 1 snapshot с `executors` без `expected_duration`.

### 4. UI использует display label как identity work plan

Файл: `web/src/lib/progress.ts:4`

`monotonicPercent` разрешает падение процента при изменении `label` и запрещает его при неизменном label. Контракт этого не гарантирует: label является display copy и может меняться на каждом обычном report, например `chapter 3 of 14 -> chapter 4 of 14`. И наоборот, `total` или supervisor patch может изменить work plan при `label: null`.

Следствие двустороннее:

- обычная смена текста позволяет старому или меньшему response уменьшить bar;
- реальное расширение плана с прежним или null label скрывается через `Math.max`.

Проблема присутствует и в implementation plan, который предписал эту эвристику. Рекомендация: добавить в engine/API явный deterministic `plan_revision` или `plan_key`, производный от patched snapshot и изменений cycle total, и использовать только его как reset signal. Тест `web/src/lib/progress.test.ts:13` сейчас закрепляет ошибочную модель и должен быть заменён.

### 5. Monotonic state переходит между разными runs

Файлы:

- `web/src/lib/RunProgress.svelte:9`
- `web/src/pages/RunView.svelte:92`
- `web/src/App.svelte:71`

`RunProgress` хранит `shown` и `lastLabel`, но не получает identity run. При навигации `run A -> run B` приложение остаётся в той же ветке `route.page === 'run'`, поэтому Svelte сохраняет экземпляры `RunView` и `RunProgress`. Если labels равны или оба null, run B с 20 percent может унаследовать 80 percent от run A.

Рекомендация: key component по `${workspace}:${run_id}` либо передать обязательный `runKey` и сбрасывать state при его изменении. Исправление следует проектировать вместе с явным `plan_revision` из предыдущего finding: run identity отвечает за смену запуска, plan identity за разрешённый reset внутри одного запуска.

## Minor findings

### 1. `label: null` не очищает старый label

Файл: `crates/apb-engine/src/progress.rs:132`

`label` обновляется только при `lbl.is_some()`. Это противоречит контракту latest `RunProgress.label` wins и оставляет устаревший текст. На каждом progress event нужно выполнять `label = lbl.clone()` и добавить тест `Some -> None`.

### 2. V20 не охватывает invalid YAML scalar types

Файлы:

- `crates/apb-core/src/schema.rs:173`
- `crates/apb-core/tests/validate_duration_test.rs:57`

Float, negative number, boolean и другие неподдерживаемые scalar values падают на schema deserialization до `check_expected_duration`, поэтому не получают обещанный structured diagnostic V20. Staged test закрепляет это расхождение со spec.

Рекомендация: сохранить invalid raw value в catch-all variant или custom deserializer, передать его validator и проверять V20 для float, negative и boolean.

### 3. Ошибка чтения event log подменяется отсутствием измерений

Файл: `crates/apb-mcp/src/tools.rs:728`

`read_all(run_dir).unwrap_or_default()` превращает I/O или corruption error в пустой event log и таблицу с `measured_seconds: null`. Это misleading fallback: calibration consumer не отличает поломку данных от реального отсутствия измерений.

Рекомендация: вернуть из `build_duration_table` typed `Result` и распространить ошибку во всех callers. Не представлять unreadable log как пустой.

### 4. Component behavior не покрыто тестами

Файл: `web/src/lib/progress.test.ts:1`

Тесты проверяют только pure helpers и не монтируют `RunProgress.svelte`. Не проверяются route identity transition, terminal hiding, waiting text, label, ARIA и refetch state. Это test gap, который позволил findings 4 и 5 пройти все frontend tests.

Рекомендация: добавить component/integration tests для required rendering states, смены run A на B и reset только при явном plan revision. Сам test gap классифицирован Minor, поскольку основные behavioral defects уже перечислены как Important.

### 5. `waiting_kind` типизирован слишком широко

Файл: `web/src/lib/types.ts:46`

API invariant допускает только `human_review`, `wait` и null, но тип объявлен как `string | null`. Использовать `'human_review' | 'wait' | null`, а в перспективе зеркалировать backend enum вместо `Option<String>`.

### 6. Статус design spec устарел

Файл: `docs/superpowers/specs/2026-07-17-run-progress-design.md:4`

Документ всё ещё содержит `approved design, pending implementation plan`, хотя plan и implementation уже staged. Обновить статус на `implementation in review` и синхронизировать status/checklist plan после прохождения gates.

## SOLID, KISS, DRY, fallback и hardcode

### SOLID

Явных нарушений dependency direction или циклических crate dependencies не найдено. Разделение `core -> engine -> mcp/server -> web` сохранено. Pure progress fold и общий transport-neutral `ProgressSummary` являются удачными решениями. Главный риск SRP находится не в размере модулей, а в смешении identity и display semantics: UI использует `label` одновременно как текст и как plan reset token.

### KISS

Backend formula и SCC подход в целом соответствуют задаче. Неявное использование scheduler `current` как identity reporter упрощает command shape ценой некорректности в sequential boundary и parallel path. Явный validated `node_id` будет чуть более подробным, но проще для reasoning и тестирования.

### DRY

Duration default и progress fold централизованы, дублирующей бизнес-формулы между MCP, HTTP и web не найдено. Нарушение возникает вокруг legacy snapshot parsing: progress и resume применяют разные правила к одному persisted format. Нужен общий compatibility parser для snapshot consumers.

### Fallback

Подтверждён один misleading fallback: `read_all(...).unwrap_or_default()` скрывает unreadable или corrupted event log как отсутствие измерений. Legacy progress, наоборот, не реализует требуемый fallback и возвращает null там, где spec обещает default-based calculation.

### Hardcode и constants

Необоснованных backend hardcodes в staged feature не найдено. `DEFAULT_TASK_SECONDS` переиспользуется корректно. UI string literals для двух waiting kinds являются ограниченным protocol vocabulary, но их тип следует сузить до union или общего generated contract. `label` нельзя использовать как скрытый protocol token.
