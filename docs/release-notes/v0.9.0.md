## apb 0.9.0: install story

This release is the first published through dist (formerly cargo-dist): the tag-push release pipeline now runs on a dist-generated GitHub Actions workflow instead of the old hand-written one, and every artifact in this release, the tarballs, the shell installer, the Homebrew formula, was produced by that pipeline. Nothing about running playbooks changes in this release; the whole scope is getting apb onto a machine and keeping it current.

### Two new install paths, plus keeping apb current

A one-line shell installer works on macOS and Linux: `curl --proto '=https' --tlsv1.2 -LsSf https://github.com/itechmeat/agentic-playbooks/releases/latest/download/apb-installer.sh | sh`. It detects OS and architecture, downloads the matching tarball, verifies its checksum, installs the `apb` binary under `CARGO_HOME`, and writes an install receipt that later powers self-update. Homebrew is now a first-class path too: `brew install itechmeat/agentic-playbooks/apb` taps `itechmeat/homebrew-agentic-playbooks`, a formula dist keeps in sync on every release.

`apb self-update` is a new subcommand built on axoupdater, the same mechanism `uv self update` uses. It reads the install receipt the shell installer wrote, checks GitHub Releases, verifies checksums, and replaces the binary in place; `apb self-update --check` reports without installing. Exit codes: `0` already up to date (or updated), `10` an update is available (check mode only), `2` the check or update failed, including when there is no install receipt to read, in which case the message names the right path instead (`brew upgrade apb` for Homebrew, rebuild from source for a source build). Self-update only works for installs made through the shell installer; it deliberately refuses to guess for the other two paths.

### apb init becomes a real questionnaire

`apb init` is now an interactive clack-style onboarding flow on a real terminal (a non-TTY run is unchanged, so scripts and CI keep working exactly as before). It asks once whether coding agents may report apb errors back after a playbook run, anonymized and filed transparently at the project's GitHub repository with no secrets and no private prompts; a yes writes a standing feedback-loop section into `CLAUDE.md` and `AGENTS.md`. The write is idempotent: a file that already has the section is left untouched, a file that exists without it gets the section appended with the right separator, and a missing file is created outright, so re-running `apb init` never duplicates or corrupts the block. Declining just skips it, with a note that the section can be added later from the README. The same run also offers the existing agent-subscriptions survey. Cancelling with Escape or Ctrl-C exits cleanly with a cancel message instead of turning into an error, and no other questionnaire failure changes `apb init`'s own exit code.

### Install docs, rewritten

`docs/INSTALL.md` and the README install section were stale, still describing prebuilt binaries and Homebrew as "planned" through v0.8.0 even though releases had shipped binaries for several versions. Both are rewritten to lead with the one-liner and Homebrew, document `apb self-update` and its check mode, and keep the manual-download and build-from-source paths as fallbacks. The build-from-source section still calls out that `apb-server` embeds the built web frontend via rust-embed, so `(cd web && bun install && bun run build)` has to run before `cargo install --path crates/apb-cli`.

### Release pipeline

The release workflow is now generated by `dist generate` from a committed dist config rather than hand-maintained. The pre-build quality gate (fmt, clippy, nextest, doc tests) still runs and still blocks publishing: it is wired in as a dist plan-stage job, so a failing gate stops the release before any target builds, not after. The per-tag release-notes convention is unchanged: `docs/release-notes/vX.Y.Z.md` is what a post-announce job applies as the GitHub Release body; dist only understands a root changelog on its own, so this file remains the source of truth for what a release says. The same four build targets ship as before (`aarch64-apple-darwin`, `x86_64-apple-darwin`, `x86_64-unknown-linux-gnu`, `x86_64-unknown-linux-musl`); there is no Windows build in this release. `dist plan` runs as a dry-run in pull request CI so drift between the dist config and the generated workflow fails a PR instead of a release.

The `apb-cli` package is renamed to `apb` for clean dist-derived artifact names. This is only user-visible through `cargo uninstall apb` replacing `cargo uninstall apb-cli`; the crate path (`crates/apb-cli`), the binary name (`apb`), and everything else about building from source are unchanged.

### Known limitations

crates.io publishing is still not set up; `cargo install --path` from a source checkout remains the only cargo-native path, since rust-embed needs `web/dist` built ahead of a registry install today. There is no Windows build or installer in this release. Self-update covers only installer-based installs; Homebrew and source installs update through their own normal paths (`brew upgrade`, rebuild) with no automatic detection of which path was used to install.
