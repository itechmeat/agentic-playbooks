## apb 0.7.0: supervised run reliability

This release closes out a field report from a real production run: a two-repo feature shipped through a 17-node playbook on a memory-constrained VPS that survived several host crashes. The run itself finished, but getting there took manual ps forensics, a node that re-ran three times after a crash, and hand-edited context files to get a note to a stuck run. Thirteen findings came out of that report, all verified against the 0.6.0 code, and every one is fixed on this branch.

### Headline fixes

Validation errors reaching MCP used to collapse to bare codes. They now carry code, severity, node, and message, one line per issue, in report order. The V13 unresolved-template error appends the full list of known namespaces (`params.*`, `nodes.<id>.output`, `nodes.<id>.report`, `nodes.<id>.review_note`, `run.instruction`, `run.context`, `run.hooks.*`) so a rejected template no longer needs bisecting to fix. V13 also now scans the `instruction` field of playbook nodes and finish-node prompts, which were previously blind spots.

`attempt_started` used to be journaled only when an attempt finished, so a crash mid-attempt left no trace in the journal and resume rewound to the last finished node, re-running completed work; this is how one node ran three times in the field. Attempts are now journaled at spawn, with the agent's pid attached, and `attempt_finished` now carries `duration_ms`. Resume reads the journal for an interrupted node (started but never finished) and restarts exactly that node; if nothing is interrupted, resume continues from the successors of the last finished node instead of re-running it. A new `run_resumed` event replaces the old `run_paused` marker, so a resumed run reads as running while it executes rather than paused forever. A resume that would do nothing (the run already succeeded, no argument given) is refused with a message naming `--from-node`.

The control cursor that tracks which supervisor commands have been applied is now persisted to disk, so notes and other control entries are applied exactly once instead of being replayed on every resume. Run context.md now leads with the run's instruction whenever one was given, so it survives lossy hand-offs through summarizing nodes and reaches every node that renders `{{run.context}}`.

Three new operational commands close gaps the field run had no answer for: `apb note <run_id> <text>` posts a note to a live run without needing an MCP session, `apb stop <run_id>` (and the matching `run_stop` MCP tool) actually interrupts an in-flight node instead of only taking effect at the next node boundary, and `apb doctor --run <id>` prints a per-run consistency report covering folded statuses, open attempts and their pid liveness, the driver and workdir-lock holders, and unapplied control entries.

MCP-started background and supervised runs, and `run_resume`, used to run on threads inside the `apb mcp` process and died with it. They now re-exec the `apb` binary as a detached driver process that survives the MCP session; `run_resume` returns an acknowledgement immediately instead of blocking until the run ends. A stop against a run whose driver process is gone finalizes the run directly rather than leaving it wedged in a running state.

Playbook edges can now carry `max_traversals`, so a cycle is legal as long as at least one edge in it is bounded. This lets a fix loop (review, fix, review again) be expressed directly instead of unrolled by hand into repeated nodes. Each traversal of a bounded edge is journaled, so loop progress survives resume exactly, and a node re-executed inside a loop skips the cached result from its previous run so later iterations see the fresh verdict.

### Supervision improvements

`run_status` gained `driver_alive` (whether the process claiming the run is still alive), per-node `node_times` (start timestamps and, for running nodes, attempt age and pid), and a `lost` node status for a node whose journaled attempt pid is no longer alive. Together with `doctor --run`, a stuck run is now a status read instead of manual ps forensics.

Fallback no longer retries an identical agent and model binding: before running a fallback step, the engine compares its resolved binding against the one that just failed and skips the step if they match, so a claude-to-claude fallback with the same profile does not burn a pointless retry.

`playbook_trial` now accepts an `instruction`, threaded through the same way `playbook_run` does, so trial runs of instruction-driven playbooks render `{{run.instruction}}` correctly.

The authoring howto gained sections on template variables (the exact accepted namespace set), human_review options and the three conditional edge types, and bounded loops with a worked fix-loop example. MCP.md documents the new tools, the detached-run lifecycle, and the `run_resume` acknowledgement shape.

### Known limitations

There is still no way to deliver a note to an in-flight one-shot agent process; one-shot CLI agents have no input channel mid-run. The practical pattern is stop, then note, then resume: interrupt the run, leave the note, and let the next drive pick it up.

Attempts carry no token or cost data. Adapters do not parse usage output yet, so run events have no cost or usage fields to report.

A node whose attempt is marked `lost` (pid no longer alive) is surfaced in `run_status` and `doctor --run`, but nothing respawns it automatically. Recovering a lost attempt is still a manual `apb stop` and `apb resume` (or `--from-node`) today.
