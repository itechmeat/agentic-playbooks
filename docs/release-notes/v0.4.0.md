## apb 0.4.0: incremental execution

Playbooks now reuse proven work. When a node already ran with the same inputs, the same profile bundle, and the same workspace state, apb can skip the execution and restore the recorded result: repeated runs of lint, review, and analysis playbooks stop paying for agent work whose inputs did not change.

### Node result cache

- Opt-in per node with `cache: auto`; nothing changes for playbooks that do not ask for it. Supported on `agent_task` and `script` nodes.
- Trust follows the project's verification model: the declaration is intent only. A result is admitted to the cache only after post-hoc verification confirms the node behaved read-only: the workspace fingerprint is unchanged after execution and every connector call the node made was from the connector's `read_only` function set. Failed verification rejects admission and the run continues normally.
- The cache key is content-addressed: node definition, rendered prompt (upstream outputs and params included), profile bundle digest, agent and model, connector digests, and the workspace fingerprint. Editing a skill, switching a model, or changing an upstream result invalidates dependents automatically.
- Workspace fingerprints are git-aware by default (HEAD plus staged, unstaged, and untracked state); a node can declare `inputs: { files: [...] }` globs for precise, file-scoped invalidation.
- Declared `outputs: { files: [...] }` become content-addressed artifacts: captured with digests on execution, restored byte-identical on a cache hit, listed in the node's `NodeFinished` event.
- Optional `ttl` on the cache declaration expires entries whose hidden inputs (for example direct network reads) may drift.
- The store lives in `.apb/cache` as plain content-addressed files with provenance records (run, playbook, node, digests); corrupted or tampered entries degrade to a miss and are dropped. A cache problem never fails a run.

### Events and UI

- New run events `node_cache_hit`, `node_cache_miss`, `node_cache_stored`, `node_cache_rejected` make every cache decision auditable in the event log; cache reuse is explicit, never disguised as a fresh execution.
- The dashboard run view marks cache-served nodes with a `cached` badge and shows cache events in the run's event list.

### Tooling

- `apb run` gains `--no-cache` (ignore the cache entirely) and `--refresh-cache` (re-execute and overwrite entries).
- New `apb cache` subcommands: `status` (records, objects, total size), `inspect <key>` (record with provenance), `prune [--older-than <dur>] [--max-size <size>]`, and `clear`.
- Validator codes V27, V28, and V29 catch cache declarations on unsupported node kinds, dead `ttl` values, and invalid `inputs`/`outputs` globs at authoring time.

### Known limitations

- Nodes running on parallel fan-out branches are not cached in this release; only the sequential execution path consults the cache.
