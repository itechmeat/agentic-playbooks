## apb 0.8.0: interactive nodes

An `agent_task` node can now be marked `interactive: true`, letting the agent ask the user a question mid-attempt instead of only reporting a finished result at the end. This closes the gap where a playbook either had to guess or fall back to a full `human_review` gate for anything that needed a quick clarification from the user while the agent kept working.

### The feature

Four new fields on `agent_task`: `interactive` (bool, default false) turns the node on; `answer_by` (`human` | `supervisor`, default `human`) decides who may answer; `question_timeout_seconds` (optional) bounds how long the node waits before falling back to `default_answer` (`answered_by: "timeout"`); `default_answer` requires `question_timeout_seconds` (validator V32), and any of the three companion fields set without `interactive: true` is rejected (validator V31). `answer_by: human` is a contract, not just a capability check: a supervising agent must relay the question to the user verbatim and relay the answer back verbatim, and an attempt to answer such a node through the supervisor-token path is refused with an error naming the relay requirement. `answer_by: supervisor` lets the supervisor answer from its own judgment. A node's `timeout_seconds` clock excludes the time a question sat pending, so a slow human reply never counts against the node's own work budget.

### Three transports, best available first

**live** injects a one-tool stdio MCP sidecar (`apb __ask-server`, tool `ask_user`) into the spawned agent; the tool call blocks until a human answers, with a progress notification every 60 seconds so a long wait never trips the host's idle timer. This release ships `live` for claude only: its stream-json output is the only one that yields a capturable session id today. **resume** has the agent print a question marker and exit; apb re-invokes it with the answer once a human replies, using the session id captured from the prior attempt. **reprompt** is the floor every agent falls back to: a fresh invocation carrying the full Q&A transcript in the prompt, working with any agent but losing in-flight state. Whichever transport is active, a running agent can also signal a question by printing the marker `<<<apb:question>>>` followed by a line of JSON (`{"question": "...", "options": [...]}`); this is how resume and reprompt recognize a question, and it works as a fallback for a live agent that prints it instead of calling the tool.

Codex, opencode, and hermes declare `resume` as their built-in default, but none of the three surveyed one-shot invocations surfaces a session id in its output today, so the engine downgrades every attempt to `reprompt` at runtime. Antigravity (agy) is `reprompt` by design; it has no way to surface a conversation id from a one-shot run. Every downgrade, `live` to `resume` or `resume` to `reprompt`, is journaled as `SupervisorAction { action: "interaction_downgraded", node, detail }`, so a supervisor or `apb doctor --run` can see exactly when and why a node fell back.

### Answer facades

`run_answer` (MCP) accepts either a `run_id` (direct human path) or a supervisor token (subject to the `answer_by` policy above). `run_status` gains `pending_question` (`{ node, question, options, answer_by, asked_at }`, null when nothing is pending) and `progress.waiting_kind: "question"`; `supervisor_wait_event` raises a wake the moment a question is asked. `apb answer <run> [--node <id>] <text>` is the CLI equivalent, `--node` only needed when more than one question is pending at once. A pending question shows up in `apb runs` and `apb doctor --run` (sanitized for terminal display), and the web UI's question panel presents the options plus a free-text field.

### Known limitations

Live transport is claude-only this release; codex, opencode, and hermes are declared `resume` but run as `reprompt` until one of them surfaces a usable session id from a headless one-shot invocation (tracked in `docs/INTERACTIVE-AGENTS.md`, including the upstream contribution plan and the still-pending verification checklist). Antigravity (agy) is `reprompt` only, with no live or resume path in sight. MCP elicitation (spec 2025-06) is not used: as of this release no surveyed agent ships it verified in headless mode, so the marker protocol remains the reprompt/resume mechanism. `script` and `condition` nodes are never interactive; only `agent_task` carries the four companion fields. If the `apb __ask-server` sidecar fails to register inside claude (a spawn or MCP-config error), the failure is silent from the node's point of view: the agent simply lacks the `ask_user` tool and, absent the marker protocol as a fallback, the node runs to completion non-interactively rather than asking anything.
