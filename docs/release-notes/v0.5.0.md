## apb 0.5.0: asana and mail connectors

Wave 2 of official connectors: task management on Asana and mailbox reading over IMAP, plus a template rendering fix that strict JSON APIs need and a flaky-test fix.

### Asana connector

The `asana` connector covers task triage end to end: `list_workspaces`, `list_projects`, `list_sections`, and `list_tasks` for browsing, `get_task`, `list_comments`, and `list_subtasks` for detail, `create_task`, `update_task`, `add_comment`, `create_subtask`, and `add_task_to_section` for writing, and `search_tasks` for a fuzzy typeahead lookup by name.

Auth is a single Personal Access Token sent as a Bearer header; workspace, project, section, and task gids are call arguments rather than account fields, so one account reaches every workspace the token can see.

Write functions send explicit nested `{"data": {...}}` bodies with typed fields: `completed` renders as a real boolean and `projects` as a real array, and an omitted optional field (for example `notes` or `due_on` on `update_task`) is dropped from the request instead of being sent as an empty string.

`list_workspaces`, `list_projects`, and `list_tasks` paginate with Asana's own cursor: read `next_page.offset` from a response and pass it back as the `offset` argument on the next call.

### IMAP connector and the silent-read guarantee

The `imap` connector reads mailboxes on Gmail, Yandex Mail, Outlook and Microsoft 365, iCloud, and any standard IMAP server through one manifest; provider differences live entirely in account configuration (host, port, auth method), not in separate connectors.

Reading a message never marks it seen. `search_messages` and `get_message` open the folder read-only with `EXAMINE` and fetch content only with `BODY.PEEK[]`, so the `\Seen` flag cannot change as a side effect of a read even on a server that would otherwise mark a message seen on fetch. This is an engine-level invariant of the new built-in `imap` function kind, not a manifest convention: `EXAMINE` has the server itself refuse flag changes during a read-only session. The only functions that change anything on the server are `mark_read` and `mark_unread`, separately grantable calls that use `SELECT` and `STORE`.

Password auth and XOAUTH2 are both supported. For providers that no longer accept password IMAP login, such as Outlook and Microsoft 365, the account config points `auth_method` at `xoauth2` and sources the access token from an external OAuth helper through the existing `{{cmd:...}}` command-sourced secret mechanism, the same pattern the GitHub connector uses for `gh auth token`. apb implements no OAuth consent flow itself.

`docs/CONNECTORS.md` and the connector's `PUBLIC.md` walk through account setup for Gmail (app password, 2-Step Verification required), Yandex Mail (enable IMAP access, then an app password), Outlook and Microsoft 365 (xoauth2 only), and iCloud (app-specific password).

`examples/playbooks/inbox-triage.yaml` demonstrates both connectors together: searching and fetching mail over `imap`, filing a follow-up task with `asana create_task`, with `mark_unread` deliberately left out of the grants.

### Template rendering: typed and optional single placeholders

Body and query templates gain a new rendering rule for the case a wave 1 whole-string interpolation could not handle: a body string leaf or a query value that consists of exactly one `{{args.<field>}}` placeholder and nothing else now renders as the typed JSON value of that argument, so booleans stay booleans, numbers stay numbers, and arrays stay arrays instead of being stringified.

When the referenced argument is absent from the call, the enclosing body object field, body array element, or query pair is dropped from the rendered request rather than erroring or sending an empty value; an argument explicitly set to `null` is treated the same as absent. Required arguments are still enforced by `args_schema` before rendering, so the drop only ever applies to genuinely optional fields. A top-level body that is a bare single placeholder still errors when its argument is absent, and mixed-content templates such as `"prefix {{args.x}}"` keep the existing string-interpolation behavior unchanged.

Existing wave 1 manifests are unaffected: their body and query arguments are all required, so the new drop rule never fires for them.

### Flaky test fix

`connector::secrets::tests::resolve_cmd_empty_output_is_error` and its sibling stub-based tests intermittently failed with ETXTBSY when a freshly written executable stub was exec'd before the write had synced to disk. The test helper now writes through `create` plus `write_all` plus `sync_all` before setting the exec bit, mirroring the fix already applied to the engine test helpers. Production code is untouched: `resolve_cmd` executes user-installed helper binaries, never files the same process just wrote.

### Known limitations

No message deletion or moves between folders, and no non-`\Seen` flag changes: this wave is silent reading plus explicit read/unread marking only. No attachment content download, `get_message` lists attachment filename, mime type, and size but never bytes. No STARTTLS for IMAP, only implicit TLS on port 993, which is what every targeted provider uses. No OAuth consent flow inside apb for either connector, xoauth2 tokens must come from an external helper. Asana's premium search API, webhooks, custom field CRUD, portfolios, goals, and the batch API are excluded; task search uses the typeahead endpoint, which works on every Asana tier but is a fuzzy name match rather than full-text search.
