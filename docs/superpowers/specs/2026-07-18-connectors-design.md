# Connectors design

Date: 2026-07-18
Status: approved for planning
Depends on: 2026-07-08-workflows-cli-design.md, 2026-07-12-agent-profiles-design.md

## 1. Purpose and scope

A connector links a playbook node and its agent to an external service (task
tracker, messenger, mail, and similar). A connector gives the node the ability
to reach the service and gives the agent visibility of that ability: a named
set of callable functions, similar in spirit to MCP but scoped to one service
and granted per node.

This story builds the connector infrastructure only:

- the connector folder format and its validation;
- account configuration (global and project scopes) and secret handling;
- node-level binding with account and function allowlists;
- run pipeline integration (permit, manifest snapshot, grants, env scrubbing);
- the call channel (`apb connector call`) with structured errors and events;
- trust gating by content digest;
- dashboard and node-form UI plus server API;
- one fake connector for tests. It never contacts any real service;
- quality-of-life additions that ride on the same machinery: dry-run calls,
  a `read_only` function class with a grant shorthand, default accounts,
  function deprecation marks, a secrets.env template helper, interim literal
  secret redaction (temporary, see 6.2), account healthchecks, a per-node
  call budget, usage stats on the dashboard, connector scaffolding, and
  automatic effect flags in the MCP catalog.

Out of scope for this story (deliberately deferred, format must not preclude
them):

- real production connectors (Jira, Telegram, Email, Outlook). They will be
  developed later and will most likely live outside this repository;
- script-backed functions (an escape hatch for non-HTTP protocols such as
  IMAP); the manifest schema stays extensible for a future `script:` function
  kind, gated by a stricter trust story;
- MCP injection into agent sessions as an alternative call transport. The
  agent-facing instruction is generated by the engine, so the transport can
  change later without touching the connector format;
- delivery of connector-bundled skills (`skills/` subfolder) into prompts;
  the folder is reserved and covered by the digest, but not delivered yet;
- a programmatic secret redaction layer over LLM-bound output. Planned as a
  separate story; this story relies on env scrubbing and file isolation,
  plus an interim literal-match redaction in the call path (section 6.2)
  that the full layer will replace.

## 2. Decisions and their rationale

| Decision | Choice | Why |
|---|---|---|
| Function execution | Declarative HTTP in `connector.yaml`, executed by apb | Secrets never leave the apb process; a marketplace connector cannot execute arbitrary code; covers Jira, Telegram Bot API, MS Graph; reuses folder, slug, digest, and trust machinery already built for skills and profiles |
| Call channel | `apb connector call` CLI invoked by the agent | Agent-agnostic (every agent can run a shell command); no long-lived transport process; run-scoped authorization via the manifest; MCP injection can be added later as a transport upgrade |
| Secrets | `{{env.VAR}}` references resolved by apb from process env and dotenv files | Convenient (plain env names, CI-friendly) while secrets stay out of git by construction and out of agent process environments via scrubbing |
| Node binding granularity | Connector ref with optional `accounts` and `functions` allowlists | Least privilege in both dimensions; both optional so the simple case stays a one-word ref |
| Trust | Tree digest of the connector folder in the existing trust store | URL templates plus secrets are an exfiltration vector, so a foreign `connector.yaml` is as dangerous as foreign code; reuses the proven digest and permit flow |
| Account pinning | Per-account digest of non-secret fields in the trust store | The account config decides where secrets are sent (`base_url`), lives in a shared committed file, and is not covered by the playbook digest; a malicious edit to a shared project config must not silently redirect secrets |
| Public page | `PUBLIC.md` with YAML frontmatter next to `connector.yaml` | Mirrors the SKILL.md and VSCode extension pattern; machine and human parts do not mix; renders anywhere |

## 3. Connector folder format

Location: `~/.apb/connectors/<name>/`. The folder name is the connector name
and must pass the existing slug validation (`[a-z0-9][a-z0-9-]*`, max 64,
`is_safe_segment`). Resolution uses the same candidate-scope pattern as skill
resolution with a single candidate (global) for now; a project scope
(`.apb/connectors/`) can be added later without format changes.

Contents:

```
~/.apb/connectors/jira/
  connector.yaml   # machine part, the only file the engine reads at run time
  PUBLIC.md        # storefront: YAML frontmatter + markdown body
  skills/          # reserved; covered by the digest, not delivered yet
```

Installation is copying the folder; removal is deleting it. Import and export
are file operations by design.

### 3.1 connector.yaml

```yaml
name: jira                # must equal the folder name
version: 0.1.0            # semver, informational for now
healthcheck: ping         # optional: function used to probe an account
auth:
  kind: header            # header | query | basic
  header: Authorization
  value_template: "Bearer {{secret.token}}"
account_fields:           # schema of one account in the config
  - name: base_url
    required: true
  - name: token
    required: true
    secret: true          # value may only be an env reference
functions:
  - name: list_issues
    description: Search issues by JQL
    read_only: true       # optional, default false; no side effects on the service
    method: GET
    url: "{{account.base_url}}/rest/api/3/search"
    query: { jql: "{{args.jql}}" }
    args_schema: { type: object, properties: { jql: { type: string } }, required: [jql] }
    timeout_sec: 30       # optional, default 30
  - name: create_issue
    description: Create an issue
    method: POST
    url: "{{account.base_url}}/rest/api/3/issue"
    body: "{{args}}"      # pass the validated args object as the JSON body
    args_schema: { ... }
  - name: ping
    description: Fake reachability check
    mock: { status: 200, body: { ok: true } }
```

Rules:

- `name` must equal the folder name; mismatch is a validation error. The
  connector name follows the same hyphen-slug rule as profiles and skills
  (`[a-z0-9][a-z0-9-]*`, max 64).
- Unknown fields are rejected (`deny_unknown_fields`), matching profile
  parsing style.
- Function names and account field names are snake_case identifiers
  (`[a-z0-9][a-z0-9_]*`, max 64), unique within the connector. Snake case
  matches API operation and template-key conventions (`{{account.base_url}}`,
  `list_issues`); the hyphen rule applies only to folder-level names.
- A function is either HTTP (`method` + `url` present) or mock (`mock`
  present), never both.
- The `auth` block shape depends on `kind`: `header` uses `header` +
  `value_template`; `query` uses `param` + `value_template`; `basic` uses
  `username_template` + `password_template`. Exactly the fields of the chosen
  kind are allowed.
- `read_only: true` marks a function as free of side effects on the service.
  The connector author is the authority for this claim; the trust gate covers
  it like everything else in the manifest. Grants can allowlist the whole
  class (section 5).
- `deprecated: "<reason>"` on a function keeps it callable but marks it in
  the generated agent instruction; the validator warns when a node allowlist
  names a deprecated function.
- `healthcheck: <function>` names a function (usually a cheap `read_only` or
  mock one) that doctor and the dashboard call per account to probe
  configuration and auth. Optional; must reference an existing function.
- Template language is minimal placeholder substitution with three
  namespaces: `{{account.<field>}}` (non-secret account fields),
  `{{args.<name>}}` (validated call arguments), `{{secret.<field>}}`
  (secret-marked account fields). No expressions, no filters, no nesting.
- `{{secret.*}}` is allowed only inside the `auth` block. Any occurrence in
  `url`, `query`, or `body` is a hard validation error. This bounds the
  exfiltration surface and makes request URLs safe to log.
- `args_schema` is a JSON Schema object; call arguments are validated against
  it before any template is rendered.
- `body: "{{args}}"` is a special form meaning the whole validated args
  object serialized as the JSON request body. Otherwise `body` is a YAML
  value whose string leaves may contain `{{account.*}}` and `{{args.*}}`
  placeholders.
- `mock` functions execute without any network. They exist for the test
  connector and for connector authoring, and are a permanent, legal part of
  the format.

### 3.2 PUBLIC.md

Frontmatter fields (all optional except `display_name` and `summary`):
`display_name`, `summary`, `tags` (list), `publisher`, `homepage`, `icon`
(relative path inside the folder). The markdown body is the long description.
The engine never reads PUBLIC.md at run time; the dashboard and any future
marketplace render it. The tree digest covers the whole folder, so editing
PUBLIC.md also drops trust, which is correct: the storefront is part of what
the user approved.

## 4. Account configuration and secrets

### 4.1 Config files

- Global: `~/.apb/connector-config/<connector>.yaml`
- Project: `<root>/.apb/connector-config/<connector>.yaml`

Both files are non-secret and safe to commit and share:

```yaml
accounts:
  - name: project-board          # slug, unique within the merged list
    base_url: https://client.atlassian.net
    token: "{{env.JIRA_PROJECT_BOARD_TOKEN}}"
```

Merge semantics: the merged account list is global accounts plus project
accounts; a project account with the same name replaces the global one
entirely; all others are additive.

An account may set `default: true`. When a node grant contains several
accounts and the call omits `--account`, the default one is used; with no
default and several granted accounts the call fails with `config` and lists
the choices. At most one default per merged list (a project default wins over
a global one); more than one at the same scope is a validation error. The
default flag participates in the account digest like any other non-secret
field.

Field values for `secret: true` account fields must be exactly one
`{{env.VAR}}` reference (a literal secret value in a config file is a
validation error). Non-secret fields are literal values.

### 4.2 Secret resolution

An `{{env.VAR}}` reference resolves at call time, in order:

1. the process environment of the resolving apb process;
2. project dotenv `<root>/.apb/secrets.env`;
3. global dotenv `~/.apb/secrets.env`.

Dotenv files are `KEY=value` lines, written atomically with mode 0600 via
`apb_core::fsutil`, and are read only by apb. When apb creates the project
`secrets.env` it appends the path to the project `.gitignore`; the validator
warns when the file exists but is not gitignored.

### 4.3 Isolation guarantees

- Secret values never appear in the run manifest, the event log, CLI output,
  or generated prompts. The manifest records only the env variable names each
  account requires.
- The engine collects every env name referenced by the connector configs of
  ALL installed connectors (both scopes, not only the connectors a run uses)
  and removes those variables from the environment of every spawned agent
  process (adapter-level scrubbing). Scrubbing the union costs nothing and
  closes the case where a run without connector X still spawns an agent that
  could read X's token from inherited env.
- `apb connector call` runs as a child of the agent and therefore inherits
  the scrubbed environment; it resolves secrets from the dotenv files itself.
  Direct user or CI invocations still work through the process environment.
- A later story adds a programmatic redaction pass over LLM-bound text as
  defense in depth.

## 5. Playbook binding

`agent_task` nodes get an optional `connectors` field. Each entry is either a
string shorthand or an object, the same two-form pattern as `SkillRef`:

```yaml
- id: triage
  kind: agent_task
  profile: dev
  connectors:
    - jira                              # everything allowed
    - name: telegram
      accounts: [team-bot]              # optional allowlist; absent = all
      functions: [send_message]         # optional allowlist; absent = all
      max_calls: 50                     # optional per-node call budget
    - name: github
      functions: read_only              # shorthand: every read_only function
```

`functions` accepts either an explicit list or the string `read_only`, which
grants exactly the functions marked `read_only: true` in the manifest. The
shorthand resolves at run start and the resolved list lands in the manifest
grant, so a later connector edit cannot widen a running grant.

`max_calls` bounds how many connector calls the node may make in one run
(across retries). It is a safety budget against a looping agent hammering a
paid API, not a rate limiter. The counter derives from the run's event log;
exceeding it returns a `permission` error naming the budget.

Playbook schema stays at version 2; the field is additive and optional, no
migration is needed.

New validator checks (codes assigned in sequence after the current last V
code):

- referenced connector is not installed;
- an allowlisted account does not exist in the merged config;
- an allowlisted function does not exist in the connector manifest;
- a connector ref name is not a valid slug;
- a configured account is missing required `account_fields` (or has a literal
  value in a secret field);
- more than one `default: true` account at the same scope;
- `functions: read_only` used with a connector that has no read-only
  functions;
- warning: a node allowlist names a deprecated function;
- warning: project `secrets.env` exists but is not covered by `.gitignore`.

## 6. Run pipeline

The flow mirrors the existing anti-TOCTOU design for profiles: one gate pass
produces a permit, the engine verifies the permit against reality, everything
after start reads only the manifest.

1. **Policy gate** (`policy::check_run`): resolves every connector referenced
   by the playbook, computes tree digests, merges account configs, computes
   per-account digests over the canonical serialization of each account's
   non-secret fields, checks trust for both connectors and accounts
   (section 7), verifies every required env name resolves (early failure
   with a message naming the account and the variable), and adds two maps to
   the `RunPermit`: `connector name -> tree digest` and
   `(connector, account) -> account digest`. The account map covers EVERY
   merged account of every connector the playbook uses, not only the
   accounts nodes are granted: any merged account is reachable through
   config-level behavior (default flags, later grant edits), so trust and
   drift detection cover the full merged set. Both maps are computed once
   and handed to the engine verbatim, never recomputed.
2. **Run start**: the engine re-verifies both permit maps against the live
   files, bidirectionally and with exact key-set match (an unexpected,
   missing, or digest-drifted entry in either direction refuses the run), copies each used `connector.yaml` into `runs/<id>/connectors/`,
   and writes into the write-once manifest: connector digests, the merged
   non-secret account fields (values, not just names), per-node grants
   (resolved account names and function names), and the required env variable
   names. Neither connector files nor account config files are read again
   after start; retry, fallback, resume, and every call use the snapshot.
   Only secret values resolve live at call time.
3. **Agent spawn**: the adapter scrubs connector env names from the child
   environment and injects a generated instruction block into the node prompt:
   the granted functions with descriptions and args schemas, the granted
   account names with their non-secret fields, and the exact call syntax.
4. **Call**: the agent runs
   `apb connector call <connector> <function> [--account <name>] --args '<json>'`
   (`--args -` reads the JSON from stdin, for large payloads). The process
   reads the run context from env variables set by the adapter (run id, node
   id), opens the run manifest, checks the grant (connector, function,
   account; `--account` may be omitted when the grant contains exactly one),
   loads the snapshotted definition and the snapshotted account fields,
   checks the node's `max_calls` budget when one is set, validates args
   against `args_schema`, resolves secrets, executes the HTTP request or the
   mock, prints a structured JSON result to stdout, and appends a call event
   to the run event log. With `--dry-run` the process stops after rendering:
   it prints the method, the pre-auth URL, and the body without executing
   anything, does not need secrets to resolve, and counts against no budget.

### 6.1 HTTP execution hardening

- The fully rendered URL must be absolute with scheme `http` or `https` and
  must not contain userinfo (`user:pass@host` is rejected). An account field
  that a function uses as a URL prefix must itself parse as an absolute
  http(s) URL (validated when the config is loaded).
- `{{args.*}}` and `{{account.*}}` substitutions into URL path segments and
  query values are percent-encoded by the renderer. Templates cannot inject
  path traversal or extra query parameters through argument values.
- Redirects are not followed. A 3xx answer is returned to the agent as a
  `service` error carrying the status; this removes the class of attacks
  where a redirect re-sends the auth header to another host.
- Response bodies are read up to a fixed cap (default 1 MiB). A longer body
  is truncated and the result is marked `"truncated": true`. If the response
  content type is JSON the body is returned parsed; otherwise it is returned
  as a string.
- The per-function `timeout_sec` bounds the whole request including connect
  time.

### 6.2 Interim literal secret redaction

TEMPORARY SOLUTION. This is a stopgap until the dedicated redaction layer
over all LLM-bound output ships as its own story; that layer replaces this
mechanism, and the implementation must carry an explicit TODO marker pointing
at it.

The `connector call` process holds the resolved secret values in memory at
the moment of the call anyway. Before printing the result to stdout and
before writing bodies to the event log, it replaces every literal occurrence
of each resolved secret value with `[redacted:<ENV_NAME>]`. This closes the
most common real leak, a service echoing the token back in a response body,
at the cost of a string scan.

Known limits, accepted for the interim: only exact literal matches are
caught; encoded, split, or transformed values pass through. The full layer
addresses those generically.

Event log: new event payload fields are added with `#[serde(default)]` only.
A call event records connector, function, account name, the request URL,
outcome code, HTTP status, and duration. The recorded URL is the one rendered
BEFORE auth injection, so `query`-kind auth never reaches the log. Request
and response bodies are recorded truncated to a fixed cap and never contain
secret values (secrets can only reach the `auth` block, which is never
logged).

## 7. Trust

The existing `TrustStore` gains two record kinds.

**Connector records**, keyed by connector name, hold the approved tree
digest of the folder. The policy gate refuses a run whose connector is
unapproved or whose digest differs from the approved one, with the same
approve-and-retry UX as profile bundles. Editing any file in the connector
folder changes the digest and drops trust.

**Account records**, keyed by `(connector, account name)`, hold the approved
digest of the account's non-secret fields in canonical serialization. The
gate refuses a run that grants a node an account that is new or whose fields
changed since approval. Rationale: the account config decides where secrets
are sent (`base_url` and similar endpoint fields), it lives in a committed,
shareable file, and it is covered by neither the playbook digest nor the
connector digest. Without pinning, one malicious commit to a shared project
config would silently redirect an auth token to an attacker host. Approving
an account is a one-time action per change, listed with the concrete field
values so the user sees what they approve.

Node grants need no separate consent: they are part of the playbook YAML and
are covered by the playbook digest.

## 8. Call results and error taxonomy

`apb connector call` always prints a single JSON document:

```json
{ "ok": true, "status": 200, "body": { ... } }
{ "ok": false, "error": { "code": "rate_limited", "message": "...", "http_status": 429, "retry_after_sec": 30 } }
```

`body` follows the rules of section 6.1: parsed JSON when the response is
JSON, a string otherwise, with `"truncated": true` set when the size cap was
hit.

Error codes:

| Code | Meaning |
|---|---|
| `config` | account or env variable not configured |
| `permission` | the node grant does not allow this connector, function, or account |
| `invalid_args` | args failed schema validation |
| `auth` | the service answered 401 or 403 |
| `not_found` | the service answered 404 |
| `rate_limited` | the service answered 429; `retry_after_sec` populated when the service provides it |
| `service` | the service answered 5xx |
| `network` | connection failure |
| `timeout` | the per-function timeout elapsed |

The HTTP-status mapping above is the default; a connector may override the
mapping for specific statuses per function when a service abuses status codes.
There are no automatic retries: the agent decides, and the structured code
plus `retry_after_sec` give it enough to decide well. Exit code is 0 when
`ok: true`, non-zero otherwise, so shell-level checks also work.

Messages inside `error.message` are machine-facing English; any user-facing
chat text about failures is produced by the agent in the user's language, per
the existing convention.

## 9. UI and server API

Dashboard, new "Connectors" page:

- list of installed connectors as cards built from PUBLIC.md frontmatter;
- a detail page rendering the markdown body, the function list from the
  manifest, per-account configuration status (configured, or missing env
  variables listed by name, never by value), and trust status for both the
  connector digest and each account (approved, changed since approval, or
  never approved);
- when the manifest declares a `healthcheck`, a per-account probe button that
  invokes it through the regular call path and shows the structured outcome
  (reachable, `auth` failure, and so on);
- usage stats aggregated from existing run event logs: calls, error rates,
  and durations per connector, function, and account over recent runs. Read
  only, no new engine state.

Node form (`agent_task`):

- a searchable combobox over installed connectors to add a binding;
- for each added connector: account checkboxes and function checkboxes, all
  checked by default (checked-all serializes as the absent-allowlist form);
- an "untrusted" badge when the connector digest is not approved.

Server API (`apb-server`):

- `GET /api/connectors` - installed connectors with frontmatter summary,
  trust status, and account config status;
- `GET /api/connectors/:name` - manifest (functions, account fields),
  rendered storefront data, merged account list (non-secret fields plus
  env-name fill status).

Playbook saving goes through the existing endpoint; `connectors` is part of
node YAML.

## 10. CLI

- `apb connector list` - installed connectors, trust and config status;
- `apb connector show <name>` - manifest summary and account status;
- `apb connector call <name> <function> [--account <a>] --args '<json>'` -
  the agent-facing call channel (section 6); also callable by users for
  debugging. `--args -` reads stdin; `--dry-run` renders without executing;
- `apb connector doctor` - checks every installed connector: manifest parses,
  digest computes, configs valid, env names resolvable, trust status of the
  connector and each account, healthcheck outcomes where declared; prints
  actionable findings;
- `apb connector env [<name>]` - prints the env variables the configured
  accounts require that do not currently resolve, as ready-to-paste
  `KEY=` template lines for `secrets.env`. Names only, never values;
- `apb connector init <name>` - scaffolds a new connector folder from an
  embedded template: a manifest with one HTTP and one mock function and a
  PUBLIC.md with filled frontmatter. Fixes the canonical authoring style.

Trust approval reuses the existing approve flow extended to connector
records.

## 11. Fake connector and test plan

Test fixture `mock-tracker` (lives in the repository test fixtures, never
contacts any real service):

- mock functions: a success case, an `auth` failure, a `rate_limited` failure
  with `retry_after_sec`;
- one HTTP function exercised against a local ephemeral HTTP server started
  by the test itself.

Coverage:

- **core units**: manifest parsing (including rejection of unknown fields,
  name mismatch, secret placement outside `auth`), config merge semantics,
  template rendering with percent-encoding, env reference validation, tree
  digest stability, account digest canonicalization (field order does not
  change the digest), URL hardening (scheme allowlist, userinfo rejection);
- **engine integration**: manifest snapshot of connectors, account fields,
  and grants; grant enforcement in `connector call` (function, account,
  node); calls read snapshotted account fields, not live configs (a config
  edited mid-run does not change where a call goes); env scrubbing of spawned
  agents covers all installed connectors; digest drift refusal at start;
  redirect refusal; response cap truncation; event log entries record the
  pre-auth URL;
- **mcp policy**: unapproved connector refused, approved digest passes, new
  or changed account refused until approved, both permit maps verified
  verbatim, early failure on unresolved env names;
- **cli**: `connector call` end to end against `mock-tracker`, `--args -`
  stdin form, `--dry-run` rendering without secrets, structured error output,
  exit codes, `connector env` template output, `connector init` scaffold
  validity (the scaffolded folder passes doctor);
- **quality-of-life paths**: `functions: read_only` resolves to the marked
  set and freezes in the manifest grant, default-account selection and the
  several-accounts-no-default failure, `max_calls` exhaustion returns
  `permission`, interim redaction replaces an echoed secret in both stdout
  and the event log, healthcheck outcome surfaces through doctor;
- **web (vitest)**: node form connector combobox and allowlist widgets,
  connectors page rendering from API fixtures.

## 12. Crate placement and MCP surface

The dependency direction stays core <- engine <- mcp, with cli and server on
top:

- **apb-core**: connector manifest schema and parsing, account config schema,
  scope merge, secret reference validation, template rendering with encoding
  rules, new validator codes, tree and account digests (reusing
  `content.rs`), trust record kinds. No async, no network.
- **apb-engine**: manifest snapshot of connectors, account fields, and
  grants; generated agent instruction; adapter env scrubbing; call execution
  (HTTP and mock), interim redaction, call events.
- **apb-mcp**: policy gate extension (`check_run` verifies connector and
  account trust, both permit maps), a `connectors_list` tool so an authoring
  agent can discover installed connectors, and automatic effect flags: a
  playbook whose nodes bind connectors is reported in `playbook_catalog`
  with a network / external-service effect derived from the YAML, so the
  driving agent discloses the blast radius before running.
- **apb-cli**: the `connector` subcommands (section 10).
- **apb-server + web**: connectors pages and API (section 9).

## 13. Known limitations

- Only HTTP and mock functions; non-HTTP protocols (IMAP) wait for the
  script escape hatch story.
- Connector definitions are global-only; project-scoped connectors are a
  possible later addition and the resolution code is shaped for it.
- Bundled `skills/` inside a connector folder are reserved but not delivered
  to prompts yet.
- Secret redaction over LLM-bound output is a separate future layer; this
  story guarantees isolation by construction (env scrubbing, auth-only secret
  templates, no secrets in manifest, log, or CLI output).
- Long-polling style functions work only within the per-function timeout;
  push-style subscriptions (webhooks) are out of scope.
- No client-side rate limiting or per-account concurrency control; parallel
  nodes calling the same account rely on the service's own limits, surfaced
  to agents as `rate_limited` errors.

## 14. Threat model

What the design defends against, and what it does not. Being explicit here
prevents the guarantees from sounding stronger than they are.

**Defended:**

- Secrets leaking into LLM context, prompts, run manifests, event logs, CLI
  output, or the repository. This is the primary goal and it holds by
  construction: secrets exist only inside short-lived apb processes, are
  referenced by name everywhere else, and are scrubbed from agent
  environments.
- A malicious connector definition (copied from a marketplace) exfiltrating
  secrets: no code execution, secrets render only into the `auth` block,
  trust gates every content change by digest.
- A malicious edit to a shared project config redirecting secrets to another
  host: account pinning (section 7) forces an explicit re-approval that shows
  the changed fields.
- Definition or config drift between the policy check and the run, and
  during the run: one-pass permit, write-once manifest snapshot, calls read
  only the snapshot.
- Redirect-based auth replay to a third host and URL injection through call
  arguments: section 6.1.

**Not defended (out of scope, by honesty):**

- A deliberately malicious agent with unrestricted shell access. Agents run
  as the same OS user, so such an agent could read `~/.apb/secrets.env`
  directly, bypassing every apb-level control. The real guards at that layer
  are the agents' own sandboxing and permission modes; apb's contribution is
  that secrets never appear in anything apb hands to the model, so a
  well-behaved agent cannot leak them by accident.
- Node identity spoofing within a run: the run and node ids that
  `apb connector call` reads from env are cooperative identification, not
  authentication. A hostile process under the same OS user gains nothing it
  could not already take directly (see the previous point), so a heavier
  token scheme would add complexity without adding security.
- A compromised external service echoing secrets back in transformed form
  (encoded, split, embedded). The interim literal redaction (section 6.2)
  catches exact echoes only; the planned redaction layer (separate story)
  addresses LLM-bound scrubbing generically.
